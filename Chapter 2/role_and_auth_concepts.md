## 1. Role vs Profile

| | Role | Profile |
|---|---|---|
| **What it is** | Business container | Technical output |
| **Created in** | PFCG | Auto-generated by PFCG |
| **Contains** | T-codes, org levels, auth objects | Auth object field values only |
| **Assigned to user** | SU01 → Roles tab | SU01 → Profiles tab |
| **Loads into buffer** | ❌ No | ✅ Yes |
| **Validity dates** | ✅ Yes — per user assignment | ❌ No |

> 💡 **Key Point:** Role is what admins manage. Profile is what the kernel loads into the User Buffer at login. The user never "has a role" at runtime — they have the **profile values** that the role generated.

## 2. Role Assignment Validity Dates

- Validity dates are set **per user per role** — NOT on the role itself
- The **same role** can have **different validity dates** for different users
- Direct profile assignment has **no date control** — always active until manually removed

```
User: BASISADMIN
─────────────────────────────────────────
Role Name      | Valid From  | Valid To
MM_PURCHASER   | 01.04.2026  | 31.12.2026   ← only for BASISADMIN

User: JOHN
─────────────────────────────────────────
Role Name      | Valid From  | Valid To
MM_PURCHASER   | 01.01.2026  | 31.03.2026   ← different dates for JOHN
```

### Role Assignment vs Direct Profile Assignment

| | Role Assignment | Direct Profile Assignment |
|---|---|---|
| **Valid From / To** | ✅ Yes — date controlled | ❌ No — always on |
| **Auto expiry** | ✅ Yes | ❌ No |
| **Audit friendly** | ✅ Yes | ❌ No |
| **Recommended** | ✅ Yes | ❌ Old approach — avoid |

> ⚠️ **Best Practice:** Always assign via Roles (SU01 → Roles tab) — never via direct profile assignment. Direct profile assignment is the old pre-PFCG approach and loses all date control.

## 3. SU22 vs SU24 vs PFCG

| Tool | Maintained by | Purpose |
|---|---|---|
| **SU22** | SAP (standard delivery) | SAP's recommendation of auth objects per T-code |
| **SU24** | Basis/Security team | Active table PFCG reads for auth object proposals |
| **PFCG** | Basis/Security team | Reads SU24 to propose auth objects when T-code is added |

```
SAP delivers SU22
(SAP's standard recommendations)
        │
        │  copy/sync
        ▼
SU24
(your active settings — what PFCG reads)
        │
        │  guides
        ▼
PFCG
(proposes auth objects when you add T-codes)
        │
        │  generates
        ▼
Role + Profile
(assigned to user → loads into buffer at login)
```

> 💡 **Key Point:** SU22 is SAP's standard. SU24 is your active settings. **PFCG reads SU24 — never SU22 directly.**

## 4. How PFCG Proposes Auth Objects

```
You add T-code to Role Menu in PFCG
        │
        ▼
PFCG reads SU24 for that T-code
        │
        ┌──────────────────────────────┐
        │                              │
SU24 has entries                SU24 has no entries
with Default = Yes              (0 entries)
        │                              │
        ▼                              ▼
PFCG proposes auth          PFCG proposes nothing ❌
objects automatically ✅    Must add manually via
                            "Manually" button in PFCG
```

### What gets automatically added regardless

When you add a T-code to the role menu, **S_TCODE** is always added automatically by PFCG — regardless of SU24 settings. This ensures the user can at least open the transaction.

## 5. SU24 Check Indicator vs Default Status

| Setting | Meaning | PFCG Behaviour |
|---|---|---|
| **Check = Yes, Default = Yes** | SAP checks it AND proposes it | Auto-proposed in PFCG ✅ |
| **Check = Yes, Default = No** | SAP checks it but won't propose | Must add manually in PFCG ⚠️ |
| **Check = No** | SAP doesn't check this object | PFCG ignores it completely ❌ |

> 💡 **Tip:** When PFCG is not proposing objects you expect — **always check SU24 first.** Either the Default flag is off, or the object is not maintained in SU24 for that T-code at all.

## 6. SU25 — Version Upgrade Tool

- After every **S/4HANA upgrade or support package** — new auth objects may be delivered by SAP into SU22
- **SU25** compares old SU22 data with new SAP delivery and helps sync SU24
- Without running SU25 after upgrades — SU24 becomes outdated and PFCG proposals become inaccurate

| Transaction | Purpose |
|---|---|
| **SU24** | Maintain auth default values for individual T-codes |
| **SU25** | Upgrade tool — sync SU22 → SU24 after upgrades/support packages |

> ⚠️ **Important:** SU25 should always be run after every S/4HANA upgrade or major support package to keep SU24 current.

## 7. Auth Objects Differ Across S/4HANA Versions

Auth objects can differ across S/4HANA versions due to:

| Reason | Details |
|---|---|
| **SAP Releases** | Each version introduces new objects — especially for CDS views, Fiori, Simplified Finance |
| **Support Packages** | Even within the same version, a support package can introduce new auth objects |
| **New Fiori objects** | Completely new auth objects in S/4HANA not present in ECC |
| **CDS View objects** | New authorization concept using CDS-based access control |

> 💡 **Best Practice:** Always check **SU24 → SAP Data** for the exact version you are working on. Never assume auth objects are identical across versions.

## 8. Fiori Authorization — 3 Layers

In S/4HANA Fiori, authorization is a **multi-layer concept** — not just one PFCG role:

```
User opens Fiori App
        │
        ▼
Layer 1 — Fiori Catalog/Group
(Frontend Role in PFCG)
Controls tile visibility on Launchpad
        │
        ▼
Layer 2 — OData/Gateway (IWFND/IWBEP namespace)
Controls OData service access at Gateway layer
Maintained in Backend Role in PFCG
        │
        ▼
Layer 3 — Backend Auth Objects
(M_*, F_*, V_* etc.)
Controls actual business data access
Maintained in Backend Role in PFCG
        │
        ▼
Layer 4 — HANA DB Privileges
Controls data access at database level
Maintained in HANA Cockpit
```

### Fiori Role Types

| Role Type | Lives on | Controls | Maintained in |
|---|---|---|---|
| **Frontend Role** | Frontend/Hub system | Tile and catalog visibility | PFCG |
| **Backend Role** | Backend system | IWFND/IWBEP objects + business auth objects | PFCG |
| **HANA Privileges** | HANA DB | Data access at DB layer | HANA Cockpit |

### Key Fiori Auth Object Namespaces

| Namespace | Purpose |
|---|---|
| **IWFND** | Gateway Foundation — OData service access at frontend/gateway |
| **IWBEP** | Backend OData service authorization |
| **SAP_FIORI_*** | Fiori catalog and group assignments |

> 💡 **Key Point:** PFCG is still the central tool for ALL role maintenance in S/4HANA — both Fiori and classic GUI. But Fiori requires maintaining **two roles** (frontend + backend) instead of just one.

## 9. Key Transactions Reference

| Transaction | Purpose |
|---|---|
| `PFCG` | Create and maintain roles |
| `SU01` | User master — assign roles/profiles, set validity dates |
| `SU22` | View SAP standard auth object recommendations per T-code |
| `SU24` | Maintain active auth object proposals per T-code |
| `SU25` | Sync SU24 after S/4HANA upgrades/support packages |
| `SU53` | View last failed authorization check for a user |
| `SU56` | View full user buffer for a user |
| `SM04` | View and terminate active user sessions |
| `AL08` | View all active users across all application servers |

## 10. Golden Rule

```
SU22 guides SU24
        │
        ▼
SU24 guides PFCG
        │
        ▼
PFCG builds the Role
        │
        ▼
Role generates the Profile
        │
        ▼
Profile loads into the User Buffer at login
        │
        ▼
Buffer is checked at every AUTHORITY-CHECK!
```

> **The complete chain — from SAP standard delivery all the way to the runtime authorization check — flows through these tools in sequence. Understanding this chain is the foundation of SAP authorization management.**
